\begin{abstract}
LunarAtlas is an open-source, reproducible data processing infrastructure that transforms publicly available Chandrayaan-3 laser-induced breakdown spectroscopy (LIBS) Level-1 (L1) products from the ISRO Planetary Data Archive into normalised, analysis-ready spectral records suitable for quantitative lunar science. The Chandrayaan-3 LE-LIBS L1 archive distributes calibrated spectra in wide-format tables where 2,094 wavelength channels (instrument range: 220--800~nm; stored channel range: 164.35--878.26~nm) appear as individual columns, with plasma and background acquisitions interleaved. To our knowledge, no publicly documented, reproducible pipeline exists to ingest and clean these products automatically. LunarAtlas implements a six-stage deterministic Python pipeline that: (i) parses PDS4 XML metadata; (ii) verifies file provenance via MD5 checksums; (iii) reshapes wide-format records into a normalised long-form relational layout $(\text{Measurement\_ID},\,\text{Wavelength\_nm}) \rightarrow \text{Intensity (a.u.)}$; (iv) identifies plasma and background shots from mission status flags; (v) applies paired background subtraction with clamping; and (vi) assigns file-unique Measurement IDs. For interactive visualisation, LunarAtlas employs an LTTB+Peaks adaptive downsampling algorithm that retains 100\% of detected spectral peaks. A versioned REST API delivers spectral data; under single-connection, no-load conditions on a commodity workstation, the measured API response latency was below 500~ms --- a hardware-specific benchmark result that may vary under concurrent or networked deployment. Software and documentation are available at: \url{https://github.com/LovekeshAnand/LunarAtlas} (archived snapshot: \url{https://doi.org/10.5281/zenodo.20771541}).
\end{abstract}

\section{Motivation and Significance}

Laser-Induced Breakdown Spectroscopy (LIBS) is a well-established technique for in-situ elemental analysis in planetary exploration. By focusing a pulsed laser onto a target surface, LIBS generates a transient plasma whose emission spectrum reveals the elemental composition of the sample. The technique has been deployed on multiple planetary missions, including ChemCam on MSL \cite{maurice2012chemcam, wiens2012chemcam}, SuperCam on Mars 2020 \cite{maurice2021supercam, wiens2021supercam}, and MarSCoDe on Tianwen-1 \cite{xu2021marscode}. In 2023, India's Chandrayaan-3 mission operated the Laser-Induced Breakdown Spectroscopy (LIBS) payload aboard the Pragyan rover, acquiring the first in-situ LIBS spectra from the lunar south polar region \cite{megala2024chandrayaan, sridhar2024libs, isro_chandrayaan3_libs}.

The Pragyan LIBS instrument employs a pulsed Yb:Er:Glass laser operating at $1535 \pm 1$~nm and records plasma emission spectra over the calibrated 220--800~nm spectral range using a holographic flat-field grating spectrometer coupled to a linear CCD detector. The archived Level-1 (L1) products contain 2,094 wavelength channels spanning 164.35--878.26~nm, extending beyond the instrument's calibrated operating range. Throughout this work, all scientific interpretation is restricted to the calibrated spectral interval. The instrument provides a spectral sampling of approximately 0.34~nm per channel with an optical resolution of $\le 1$~nm. These datasets are publicly distributed through the ISRO Planetary Data Archive (PRADAN) as Planetary Data System Version 4 (PDS4) archives \cite{isro_chandrayaan3_libs, pds4_standard}.

Although the archive ensures long-term preservation and public accessibility, the delivered L1 products are not directly suitable for reproducible computational analysis. Several characteristics of the archive require mission-specific preprocessing before the spectra can be queried, analysed, or visualised efficiently.
\begin{enumerate}
    \item \emph{Interactive Visualisation:} Each spectrum contains 2,094 sampled wavelength channels. Efficient web-based visualisation therefore benefits from adaptive downsampling techniques that preserve diagnostically important emission peaks while reducing rendering overhead.
    \item \emph{Tabular Reshaping:} Each spectrum is distributed as a wide-format table in which all 2,094 wavelength channels are stored as individual columns. This representation complicates relational database normalisation, prevents efficient wavelength-based queries, and limits scalable downstream analysis.
    \item \emph{Interleaved Acquisitions:} Plasma spectra and corresponding dark-field background measurements are interleaved within each L1 product according to the PRADAN acquisition sequence. Recovering cleaned plasma spectra requires parsing mission status flags and correctly pairing each plasma acquisition with its corresponding background measurement.
    \item \emph{Background Suppression:} Quantitative analysis requires subtracting the paired background spectrum from each plasma spectrum. Negative intensities produced after subtraction are clamped to zero to suppress non-physical values arising from detector noise and measurement uncertainty.
\end{enumerate}

To our knowledge, no publicly documented open-source software currently provides an end-to-end processing workflow specifically for Chandrayaan-3 LIBS Level-1 products. Table~\ref{table:capability_comparison} compares LunarAtlas with existing general-purpose and related software packages: \emph{pds4\_tools} \cite{pds4_tools_ref} and \emph{libpyhat} \cite{anderson2019libpyhat}.

While \emph{pds4\_tools} successfully parses the PDS4 XML label and locates the associated CSV via byte-offset references, it delivers the spectral payload as an unstructured delimited table with no wavelength-to-intensity axis construction. The 2,094 \texttt{Response\_Count} repetitions defined in the XML label carry no wavelength calibration; those values exist only as column headers in the CSV itself. Consequently, \emph{pds4\_tools} provides no facility for reconstructing the calibrated spectral axis, structuring the eight interleaved acquisitions into a (record $\times$ wavelength) matrix, or applying any LIBS-specific preprocessing steps.

Similarly, while \emph{libpyhat} provides documented capabilities for spectral preprocessing, normalisation, baseline removal, dimensionality reduction, and machine-learning workflows, its ingestion architecture is structurally incompatible with Chandrayaan-3 LE-LIBS L1 products. The \texttt{SpectralData} constructor requires input as a pandas DataFrame carrying a two-level MultiIndex with mandatory top-level labels \texttt{wvl}, \texttt{meta}, or \texttt{comp}. The raw L1 CSV carries a flat single-row header of the form \texttt{['Time', 'Count', 'Reset', 'Fire', '164.35', ..., '878.26']}. Because no column in this header is a tuple, the constructor's format-gate drops every column and raises an error before any processing method can be reached, resulting in a hard crash at object instantiation rather than a graceful degradation. Beyond this format incompatibility, \emph{libpyhat} provides no facility for parsing PDS4 XML labels, interpreting Force Reset Status ($F$) and Laser Fire Status ($L$) flags to distinguish plasma from background acquisitions, or pairing and subtracting interleaved shots. Thus, \emph{libpyhat} operates as a processing and analysis engine for spectra that have already been ingested and cleaned; LunarAtlas fills the upstream gap between raw ISRO PDS4 delivery and the point where analytical toolkits can be applied.

LunarAtlas addresses this software gap by providing an open-source, deterministic, and reproducible processing infrastructure that automates PDS4 metadata parsing, archive integrity verification, plasma--background pairing, background subtraction, relational database ingestion, REST API generation, and interactive spectral visualisation. By transforming archived Chandrayaan-3 LIBS products into analysis-ready spectral records, LunarAtlas enables reproducible downstream scientific workflows while eliminating the need for custom mission-specific preprocessing software.

\section{Software description}

\subsection{Software Architecture}

LunarAtlas adopts a three-tier architecture. The overall system flow is shown in Figure~\ref{fig:architecture}. The layers consist of:
\begin{itemize}
    \item \textbf{Data Layer:} A PostgreSQL database \cite{postgresql} stores mission metadata, observation provenance, and processed spectra. Processed values are stored in a normalised long-form layout, where each record represents a single channel value.
    \item \textbf{Logic Layer:} Implemented in Python~3.11 and FastAPI \cite{fastapi}, the logic layer manages NIST atomic spectra matching, adaptive downsampling, and API query routing. All numerical reductions---including spectral normalisation, dark subtraction, and peak integration---are executed server-side in Python to prevent floating-point divergence that would arise from equivalent operations in JavaScript. Result caching is handled via Redis \cite{redis}, which stores recently served downsampled payloads using a Least Recently Used (LRU) eviction policy with a 1-hour Time-To-Live (TTL); keys expire automatically after this window, eliminating the need for manual cache flushing. Since the underlying Level-1 observation records are static, cache invalidation across software versions is managed via deterministically constructed version-prefixed keys of the form \texttt{lunaratlas\_v\{n\}:\{endpoint\}:\{parameter\_hash\}}, where \texttt{n} is the software version integer and \texttt{parameter\_hash} is an MD5 digest of the query parameters. Bumping \texttt{n} upon deployment renders all prior-version keys unreachable without affecting other Redis tenants on the same instance; orphaned keys from retired versions are removed on a scheduled basis using \texttt{redis-cli --scan --pattern "lunaratlas\_v\{n\}:*"} | \texttt{xargs redis-cli del}.
    \item \textbf{Presentation Layer:} A browser-based React client \cite{react} renders interactive spectral plots. To ensure scientific integrity, all data reduction (including background subtraction, non-negativity clamping, and LTTB+Peaks downsampling) remains server-side. This architecture prevents floating-point arithmetic divergence between browser-based JavaScript engines and Python-based scientific libraries, ensuring that the visualised spectra match the ingested database records bit-for-bit.
\end{itemize}



Dependency versions are pinned in \texttt{requirements.txt} (Python~3.11) and \texttt{package.json} (Node.js~18). A multi-stage Docker image is provided at the repository root (\texttt{Dockerfile}), which builds the Vite frontend and the FastAPI backend together for simplified single-container deployment. Separate \texttt{Dockerfile.backend} and \texttt{Dockerfile.frontend} images are also provided for independent service deployment. A tagged release snapshot of LunarAtlas~v1.0.0 is archived at Zenodo~\cite{lunaratlas_zenodo} to satisfy reproducibility requirements. To ensure computational reproducibility and allow offline verification without PRADAN archive access, a comprehensive unit test suite is included in the repository under the \texttt{tests/} directory. Running \texttt{pytest tests/ -v} executes automated smoke and unit tests against synthetic input signals (e.g., simulated plasma/background spectrum pairs and baseline noise). These tests validate the core pipeline mathematics, including non-negativity clamping, paired background subtraction, LTTB downsampling ratios, and NIST peak-lock union logic.

\subsection{Data Processing and Cleaning Pipeline}

The processing pipeline is implemented in Python across two primary modules: \texttt{step2\_process\_l1\_data.py} (ingestion, pairing, cleaning) and \texttt{step3\_graph\_plotting.py} (QA plot generation). The pipeline is fully deterministic and executes in six sequential stages to transform raw PDS4 Level-1 (L1) products into cleaned relational records:

\begin{enumerate}
    \item \textbf{PDS4 Label Parsing:} The pipeline parses the accompanying PDS4 XML label to extract logical identifiers, observation start/stop times, physical units, and instrument metadata. The parser follows PDS4 field semantics \cite{pds4_standard} to locate mandatory fields including \texttt{pds:logical\_identifier}, \texttt{pds:start\_date\_time}, and \texttt{pds:stop\_date\_time}; labels from which these fields cannot be extracted are skipped with a warning.
    \item \textbf{File Provenance Verification:} A 128-bit MD5 checksum is computed over the raw L1 CSV bytes before any processing begins. MD5 is used for file integrity verification in a trusted archive context, not for cryptographic security. The digest is stored alongside ingestion timestamps and algorithm version strings, establishing a traceable audit trail. The pipeline is fully deterministic: re-running on the same input file produces bit-identical output records.
    \item \textbf{Wide-to-Long Reshaping:} Spectra are distributed as wide CSV tables in which each wavelength channel (164.35--878.26~nm; 2,094 channels total) is represented as a separate column \cite{numpy, pandas}. The pipeline reshapes each row into a normalised long-form layout, as defined by the relation in Equation~\eqref{eq:reshape}:
    \begin{equation}
        \label{eq:reshape}
        (\text{Measurement\_ID}, \text{Wavelength\_nm}) \rightarrow \text{Intensity (a.u.)}
    \end{equation}
    This relational representation eliminates position-based column assumptions and supports efficient range-based queries.



    \item \textbf{Plasma-Background Identification:} The pipeline parses Force Reset Status ($F$) and Laser Fire Status ($L$) metadata flags to identify background ($B$) and plasma ($P$) shots according to the boolean logic in Equation~\eqref{eq:flags}:
    \begin{equation}
        \label{eq:flags}
        B: (F = 1) \land (L = 0) \quad \text{and} \quad P: (F = 0) \land (L = 1)
    \end{equation}
    Background association relies on acquisition order preserved in the L1 CSV. Each plasma acquisition is associated with the most recently observed valid background acquisition, which is ordered per the PRADAN file specification. Acquisitions satisfying neither condition (e.g., $(F{=}0, L{=}0)$ or $(F{=}1, L{=}1)$) are considered anomalous and excluded. When consecutive plasma shots appear without an intervening background, each is subtracted from the most recently seen background acquisition. If a plasma shot occurs before any background has been observed in the file (e.g., at a file boundary), it is retained as the raw un-subtracted spectrum. These pairing metrics and subtraction status flags are logged in the pipeline's intermediate processed CSV files for quality auditing, while all parsed spectral intensity records are successfully ingested into the database.
    \item \textbf{Background Subtraction and Clamping:} Each plasma shot $P_i$ is paired with its most recently preceding background shot $B_i$, under the assumption of measurement stationarity within a single acquisition cycle. This assumption is justified by the Pragyan LIBS instrument's rapid acquisition cadence (sub-second intervals within a single command sequence) during which ambient temperature, detector dark current, and surface characteristics remain constant \cite{sridhar2024libs, isro_chandrayaan3_libs}. The clean spectrum is computed channel-wise according to Equation~\eqref{eq:clamping}:
    \begin{equation}
        \label{eq:clamping}
        I_{clean}(\lambda) = \max(0, I_{P_i}(\lambda) - I_{B_i}(\lambda))
    \end{equation}
    Intensity values represent detector response arbitrary units (a.u.) and must be non-negative; negative values arising from noise fluctuations are set to exactly zero. Acquisitions in which more than 90\% of channels are clamped to zero are flagged with a quality warning (recorded in the processing metrics log) to indicate a weak or non-physical plasma event, while the resulting values are still stored in the database.
    \item \textbf{Measurement ID Generation:} To resolve timestamp collisions (when multiple acquisitions share a single clock second), the pipeline assigns a sequential, file-unique Measurement ID. IDs are unique within a file; during database ingestion, a globally unique identifier is constructed by combining the observation-specific file identifier with the local Measurement ID, enabling unambiguous retrieval of spectra across the entire archive. This ID groups the 2,094 wavelength-intensity records into a single logical entity.
\end{enumerate}




\subsubsection{Justification of Paired Background Subtraction}
Paired subtraction was selected over model-based baseline methods for three reasons: (i) the LE-LIBS acquisition protocol provides temporally matched background measurements, making paired subtraction directly applicable without additional modelling assumptions \cite{sridhar2024libs}; (ii) the operation is fully deterministic and parameter-free, satisfying the reproducibility requirements of the infrastructure layer; and (iii) downstream users may apply further denoising (e.g., ALS \cite{eilers2003perfect} or Savitzky-Golay \cite{savitzky1964}) without the cleaning tier imposing irreversible smoothing choices. \subsubsection{LTTB+Peaks Adaptive Downsampling}
A full-resolution LE-LIBS spectrum contains 2,094 channels. Rendering thousands of spectra simultaneously in a browser-based visualisation console creates significant client-side overhead. Downsampling is therefore required for interactive visualisation while preserving diagnostically important spectral features.

LunarAtlas downsamples the data to a target count $N(k) = \min(2094, \lceil 2094 \cdot 2^{-k} \rceil)$, where $k$ is the zoom level ($k=0$ returns the full 2,094 points). The Largest Triangle Three Buckets (LTTB) algorithm \cite{steinarsson2013lttb} partitions the spectrum into buckets and selects points that maximise the triangle area formed with adjacent buckets. For a candidate point $P_c$ in bucket $j$, where $i_a, i_c, i_b$ denote the channel indices and $I_a, I_c, I_b$ their respective intensities, the signed triangle area is computed according to Equation~\eqref{eq:lttb}:
\begin{equation}
    \label{eq:lttb}
    A(P_a, P_c, P_b) = \frac{1}{2} |i_a(I_c - I_b) + i_c(I_b - I_a) + i_b(I_a - I_c)|
\end{equation}
The candidate maximising $|A|$ is selected. However, geometric downsampling algorithms like LTTB do not guarantee peak preservation.

To prevent the loss of narrow diagnostic emission lines, LunarAtlas implements a Targeted Peak Preservation (Peak-Union) step. The Peak-Union procedure operates as follows:
\begin{enumerate}
  \item Retrieve the set of curated NIST reference wavelengths $P_{\text{NIST}}$ corresponding to target elements under investigation (e.g. Ca II at 393.366~nm, Mg II at 279.553~nm, etc.).
  \item Run standard LTTB on the full spectrum to produce the downsampled index set $S_{\text{LTTB}}$.
  \item Compute $S_{\text{final}} = S_{\text{LTTB}} \cup \{i_p : p \in P_{\text{NIST}}\}$, where $i_p$ is the nearest channel index to each reference wavelength.
  \item Sort $S_{\text{final}}$ by wavelength index.
\end{enumerate}
The output size is therefore $|S_{\text{final}}| \leq |S_{\text{LTTB}}| + |P_{\text{NIST}}|$, with equality when no target peaks fall on LTTB-selected indices.

The peak retention rate measures the percentage of originally detected spectral peaks that are preserved in the downsampled subset (either exactly or at adjacent channels).

\section{Illustrative examples}

A typical workflow involves cloning the repository, installing dependencies, and executing the multi-stage ingestion pipeline. The live deployment is accessible at \url{https://www.lunaratlas.in} and the versioned REST API is served at \url{https://api.lunaratlas.in}. These live services are hosted for interactive demonstration purposes; the primary, citable, and reproducible artifact of this work is the versioned codebase archived at Zenodo \cite{lunaratlas_zenodo}. The following CLI commands illustrate local environment setup, pipeline execution, and server startup:

\begin{lstlisting}[language=bash, caption=Environment setup, pipeline execution, and server boot commands]
# Install server and client dependencies
pip install -r core/server/requirements.txt
cd core/client && npm install && cd ../..

# Execute the 6-stage data processing and ingestion pipeline
cd Pipeline && ./run_pipeline.sh

# Start the FastAPI backend API server
cd core/server && uvicorn app.main:app --host 0.0.0.0 --port 8000

# Start the React client developer server (in a separate shell)
cd core/client && npm run dev
\end{lstlisting}

Once the services are active, the REST API can be queried programmatically. The backend server exposes several endpoints: (i) \texttt{/api/v1/spectrum} which retrieves clean spectral curves with adaptive LTTB+Peaks downsampling and optional denoising filters (ALS, Savitzky-Golay); (ii) public index queries such as \texttt{/api/v1/public/observations} and \texttt{/api/v1/public/measurements} which expose PDS4 metadata and physical parameters (laser energy, pulses) filtered by date or identifier; and (iii) status monitors such as \texttt{/api/v1/health}.

The following script queries the public API to retrieve a peak-locked downsampled spectrum for a processed measurement:

\begin{lstlisting}[language=Python, caption=Python client request for downsampled spectra]
import requests

url = "https://api.lunaratlas.in/api/v1/spectrum"
params = {
    "measurement_id": "1",  # clean Measurement ID
    "lambda_min": 240.0,
    "lambda_max": 450.0,
    "zoom_level": 2,
    "als": True,
    "savgol": True
}

response = requests.get(url, params=params)
data = response.json()

# Process spectrum points (DownsampledBucket or SpectralDataPoint objects)
for pt in data["data"][:5]:
    if "wavelength_nm" in pt:
        print(f"Wavelength: {pt['wavelength_nm']} nm, Cleaned: {pt['intensity']} a.u.")
    else:
        print(f"Bucket: {pt['lambda_min']} to {pt['lambda_max']} nm")
\end{lstlisting}

\section{Impact}

\textit{Scope note.} All quantitative results in this section constitute a single worked example performed on the Chandrayaan-3 LE-LIBS Level-1 product \texttt{ch3\_lib\_002\_20230825T104221\_00\_l1}. The intent is to demonstrate that the infrastructure produces physically plausible and internally consistent outputs on a representative input file. Systematic generalisation across the full PRADAN archive has not been tested; the numerical values should therefore not be interpreted as population-level benchmarks.

\subsection{Cleaning Pipeline Validation}

All experimental evaluations and benchmarks in this section were conducted on the Chandrayaan-3 Pragyan Rover LE-LIBS Level-1 (L1) calibrated product \texttt{ch3\_lib\_002\_20230825T104221\_00\_l1} (corresponding to raw file \texttt{ch3\_lib\_002\_20230825T104221\_00\_l1.csv} containing 8 interleaved acquisitions, representing 4 background and 4 plasma measurements).

The LunarAtlas cleaning stage applies two preprocessing operations: (i) subtraction of the most recently acquired background spectrum preceding each plasma shot, and (ii) non-negativity enforcement through zero-clamping. Background subtraction is a standard spectroscopic preprocessing technique used to remove instrumental baseline and continuum contributions prior to spectral analysis \cite{cremers2006libs, chen2023automatic}.

The cleaning pipeline was evaluated on a 20~nm validation window (560--580~nm), a spectral region free of major expected lunar regolith emission lines (such as strong transitions of Fe, Ti, Ca, Al, or Mg) and representative of the instrumental continuum baseline, as validated against the NIST ASD reference database \cite{nist_asd} and standard LIBS regolith references \cite{cremers2006libs, sridhar2024libs}. In this window, the raw L1 spectrum for Measurement ID~3 had a mean intensity of $\approx1{,}387$~a.u. and a standard deviation of $\approx54$~a.u. In contrast, the cleaned spectrum had a mean intensity of $\approx444$~a.u. and a standard deviation of $\approx71$~a.u. This yields a baseline suppression factor, defined in Equation~\eqref{eq:suppression}:
\begin{equation}
    \label{eq:suppression}
    \text{Baseline suppression} = \bar{I}_{\mathrm{raw}} / \bar{I}_{\mathrm{clean}} \approx 3.12
\end{equation}
The $3.12\times$ suppression demonstrates the removal of the background continuum (Figure~\ref{fig:pre_post_overlay}). The standard deviation increase by a factor of $\approx1.31$ is consistent with noise propagation from paired subtraction \cite{bevington2003data}, closely matching the theoretical $\sqrt{2} \approx 1.41$ noise propagation factor resulting from the subtraction of two independent signals. The small discrepancy (7.7\% below the theoretical limit) is expected in practice due to: (i) minor positive correlation in detector read noise across sequential CCD readouts, and (ii) statistical uncertainty over the relatively narrow 20~nm evaluation window. Prior to clamping, approximately 0.5\% of all spectral channels for Measurement ID~3 contained negative intensity values due to statistical fluctuations. Since detector response intensity values (a.u.) are physically non-negative quantities, all negative values were clamped to zero. Table~\ref{table:ablation} summarises the pipeline metrics at each stage of the cleaning process.

\subsection{Downsampling and Peak Preservation}

Evaluation of the downsampling algorithms was performed on valid plasma measurements (IDs 3 and 4) which exhibit clear emission features. Measurements ID~3 and ID~4 were selected because they contain sufficient spectral peaks for quantitative evaluation, whereas low-signal acquisitions (e.g., ID~2) do not permit meaningful peak-preservation analysis. Weak or failed plasma acquisitions (IDs 1 and 2, characterised by $>22$\% clamped channels) were excluded for this reason.

Peak detection was performed using \texttt{scipy.signal.find\_peaks} with a prominence threshold of $3\sigma_{\text{noise}}$ (where $\sigma_{\text{noise}}$ is the population standard deviation, $\mathit{ddof}{=}0$, in the 560--580~nm continuum window) and a minimum inter-peak distance of 5 channels. This yielded 76 local maxima in Measurement ID~3 and 91 in Measurement ID~4, with prominence thresholds of $3\sigma = 212.28$~a.u. and $140.65$~a.u. respectively.

The LTTB+Peaks downsampling algorithm was compared against other decimation algorithms at 10\% data density, as summarized in Table~\ref{table:downsampling}. RMSE quantifies global spectral shape fidelity---how well the downsampled envelope approximates the full-resolution spectrum. Peak retention quantifies feature-level fidelity---whether diagnostically important emission features survive decimation. Both metrics are complementary: an algorithm may achieve low RMSE while discarding narrow peaks, or preserve all peaks while distorting the spectral envelope. Uniform decimation achieves the lowest latency but retains only 13.2\% of detected peaks, as spectral emission lines are not uniformly distributed across the wavelength axis. Max-binning achieves high peak retention (96.1\%) by selecting the maximum value within each bin, but incurs a higher RMSE (121.24~a.u.) because bin maxima selection distorts the spectral envelope between peaks. While standard LTTB achieves a lower RMSE (90.10~a.u.), it only retains 52.6\% of local maxima. The addition of the Peak-Union step in LTTB+Peaks achieves 100\% retention of detected peaks with negligible RMSE penalty (92.24~a.u.). The RMSE difference between standard LTTB (90.10~a.u.) and LTTB+Peaks (92.24~a.u.) is 2.14~a.u.\ (2.4\% relative), computed over 209 of 2,094 retained points from Measurement ID~3. As this evaluation is limited to a single spectrum, the difference should be treated as illustrative rather than statistically conclusive; it is presented here to characterise algorithm behaviour under the evaluated conditions.

To validate peak preservation from both an engineering perspective (all detected peaks) and a spectroscopy-oriented perspective (NIST reference lines), Table~\ref{table:downsampling} reports both local maxima and NIST-matched peak retention rates. A matching tolerance of $\pm0.5$~nm was adopted to accommodate the spectral resolution of the Level-1 dataset and minor wavelength calibration uncertainties. Out of 76 detected local maxima in Measurement ID~3, 6 peaks matched NIST reference wavelengths within this tolerance: Fe~I (274.86~nm), Ca~II (393.33~nm), Ca~II (396.94~nm), Mg~I (517.25~nm), Na~I (589.15~nm), and O~I (777.67~nm). The standard LTTB algorithm missed 3 of these 6 NIST-matched lines (achieving 50.0\% NIST retention), whereas LTTB+Peaks preserved all 6 NIST-matched lines (100.0\% retention) by explicitly appending detected peak indices to the downsampled set.

To demonstrate that this retention holds across different target compression levels, Table~\ref{table:density_sweep} displays peak retention as a function of downsampling density (from 5\% to 50\% points). LTTB+Peaks consistently achieves 100.0\% retention across all tested densities. Standard LTTB retention degrades to 39.5\% at 5\% density. Because the downsampling algorithm explicitly appends the original peak indices, the retained peaks exhibit zero measured position and amplitude error under all decimation ratios.

\subsection{NIST-Assisted Spectral Annotation}

LunarAtlas integrates reference wavelengths from the NIST Atomic Spectra Database (ASD) \cite{nist_asd} to support interactive spectral exploration and candidate wavelength matching. Detected local maxima are compared against NIST reference wavelengths using a $\pm0.5$~nm tolerance, adopted to accommodate the spectral resolution of the Level-1 dataset and minor wavelength calibration uncertainties. The LE-LIBS instrument has a stated optical resolution of $\leq$1~nm; the adopted $\pm0.5$~nm tolerance therefore corresponds to half a resolution element --- a relatively permissive criterion. Seven peaks matching within this tolerance out of 76 detected (9.2\%) is a modest match rate and should be interpreted as exploratory candidate identification only. Definitive elemental assignment requires corroborating evidence from additional spectral lines of the same species, line ratio analysis, and plasma temperature constraints.

\textit{Matching criterion.} A peak at $\lambda_{\mathrm{obs}}$ is a candidate match if and only if: (i) it was identified as a local maximum under the $3\sigma$ prominence threshold ($\sigma = 70.76$~a.u., threshold $= 212.28$~a.u.), and (ii) $|\lambda_{\mathrm{obs}} - \lambda_{\mathrm{NIST}}| \leq 0.5$~nm. Channels with intensity above $3\sigma$ but sub-threshold prominence --- residing within broad elevated continuum regions rather than constituting resolved emission features --- do not satisfy condition~(i) and are excluded from programmatic matching. They are rendered as reference overlays in the interface but are explicitly distinguished from candidate peak matches.

Applying both conditions to the 76 detected local maxima in Measurement ID~3, seven peaks satisfy the matching criterion (Table~\ref{table:elements}). Six additional NIST lines --- Mg~I (285.21~nm), Si~I (288.16~nm), Al~I (309.27~nm), Fe~I (404.58~nm), Ca~I (422.67~nm), and Mg~I (518.36~nm) --- have intensities of 387--572~a.u. but sub-threshold prominence and are therefore visual overlays only. The H~I Balmer-$\alpha$ line (NIST: 656.28~nm) corresponds to a detected peak at 657.09~nm (prominence 221.0~a.u.) but its offset of 0.81~nm exceeds the $\pm0.5$~nm criterion; it is retained as a flagged near-match with the offset explicitly displayed.

\section{Conclusions}

LunarAtlas provides an open-source, reproducible data processing and database infrastructure for Chandrayaan-3 LE-LIBS Level-1 observations. The pipeline implements a deterministic processing chain that resolves the challenges of tabular PDS4 parsing, interleaved acquisition pairing, and physically motivated background cleaning---validated through quantitative evaluation of cleaning, peak preservation, and API performance. The peak-locked LTTB+Peaks downsampling algorithm achieved 100\% peak retention across all evaluated spectra and density levels (5--50\%, evaluated over the two valid plasma acquisitions in the reference product) during visualisation while maintaining sub-500~ms response times on commodity hardware. By establishing an auditable, versioned reference archive for Chandrayaan-3 LIBS data, LunarAtlas supports quantitative lunar science and offers a template architecture applicable to future planetary LIBS mission archives. The system is publicly accessible at \url{https://www.lunaratlas.in}, with the REST API at \url{https://api.lunaratlas.in}.

\paragraph{Limitations.} Benchmarks and evaluations in Section~4 were performed on a single representative reference product containing four cleaned spectral measurements, and are therefore presented as a worked example rather than a generalised benchmark; statistical claims about inter-acquisition variability should therefore be interpreted as software validation rather than a comprehensive analysis of lunar LIBS measurements. No Level-2 calibration is implemented --- LunarAtlas provides analysis-ready L1 records as input to such workflows. The system has not been benchmarked under concurrent multi-user load.

\paragraph{Future work.} The most scientifically significant open direction is Level-2 calibration: translating cleaned L1 intensity records into quantitative elemental abundance estimates requires plasma temperature modelling, matrix effect correction, and validation against certified geological reference standards. On the infrastructure side, horizontal scaling for concurrent query support remains an open engineering task as community adoption grows. More broadly, LunarAtlas is intended as a template architecture for future planetary LIBS archives; its ingestion and cleaning abstractions are designed to generalise to forthcoming missions where in-situ LIBS will remain a primary geochemical tool.
