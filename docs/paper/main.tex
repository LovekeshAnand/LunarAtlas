\begin{abstract}
LunarAtlas is an open-source, reproducible data processing infrastructure that transforms publicly available Chandrayaan-3 laser-induced breakdown spectroscopy (LIBS) Level-1 (L1) products from the ISRO Planetary Data Archive into normalised, analysis-ready spectral records suitable for quantitative lunar science. The Chandrayaan-3 LE-LIBS L1 archive distributes calibrated spectra in wide-format tables where 2,094 wavelength channels (instrument range: 220--800~nm; stored channel range: 164.35--878.26~nm) appear as individual columns, with plasma and background acquisitions interleaved. To our knowledge, no publicly documented, reproducible pipeline exists to ingest and clean these products automatically. LunarAtlas implements a six-stage deterministic Python pipeline that: (i) parses PDS4 XML metadata; (ii) verifies file provenance via MD5 checksums; (iii) reshapes wide-format records into a normalised long-form relational layout $(\text{Measurement\_ID},\,\text{Wavelength\_nm}) \rightarrow \text{Intensity (a.u.)}$; (iv) identifies plasma and background shots from mission status flags; (v) applies paired background subtraction with clamping; and (vi) assigns file-unique Measurement IDs. For interactive visualisation, LunarAtlas employs an LTTB+Peaks adaptive downsampling algorithm that retains 100\% of detected spectral peaks. A versioned REST API delivers spectral data with API response latency below 500~ms on a single-node workstation. Software and documentation are available at: \url{https://github.com/LovekeshAnand/LunarAtlas}.
\end{abstract}

\section{Motivation and Significance}

Laser-Induced Breakdown Spectroscopy (LIBS) is a well-established technique for in-situ elemental analysis in planetary exploration. By focusing a pulsed laser onto a target surface, LIBS generates a transient plasma whose emission spectrum reveals the elemental composition of the sample. The technique has been deployed on multiple planetary missions, including ChemCam on MSL \cite{maurice2012chemcam, wiens2012chemcam}, SuperCam on Mars 2020 \cite{maurice2021supercam, wiens2021supercam}, and MarSCoDe on Tianwen-1 \cite{xu2021marscode}. In 2023, India's Chandrayaan-3 mission operated the Laser-Induced Breakdown Spectroscopy (LIBS) payload aboard the Pragyan rover, acquiring the first in-situ LIBS spectra from the lunar south polar region \cite{megala2024chandrayaan, sridhar2024libs, isro_chandrayaan3_libs}.

The Pragyan LIBS instrument employs a pulsed Yb:Er:Glass laser operating at $1535 \pm 1$~nm and records plasma emission spectra over the calibrated 220--800~nm spectral range using a holographic flat-field grating spectrometer coupled to a linear CCD detector. The archived Level-1 (L1) products contain 2,094 wavelength channels spanning 164.35--878.26~nm, extending beyond the instrument's calibrated operating range. Throughout this work, all scientific interpretation is restricted to the calibrated spectral interval. The instrument provides a spectral sampling of approximately 0.34~nm per channel with an optical resolution of $\le 1$~nm. These datasets are publicly distributed through the ISRO Planetary Data Archive (PRADAN) as Planetary Data System Version 4 (PDS4) archives \cite{isro_chandrayaan3_libs, pds4_standard}.

Although the archive ensures long-term preservation and public accessibility, the delivered L1 products are not directly suitable for reproducible computational analysis. Several characteristics of the archive require mission-specific preprocessing before the spectra can be queried, analysed, or visualised efficiently.
\begin{enumerate}
    \item \emph{Tabular Reshaping:} Each spectrum is distributed as a wide-format table in which all 2,094 wavelength channels are stored as individual columns. This representation complicates relational database normalisation, prevents efficient wavelength-based queries, and limits scalable downstream analysis.
    \item \emph{Interleaved Acquisitions:} Plasma spectra and corresponding dark-field background measurements are interleaved within each L1 product according to the PRADAN acquisition sequence. Recovering cleaned plasma spectra requires parsing mission status flags and correctly pairing each plasma acquisition with its corresponding background measurement.
    \item \emph{Background Suppression:} Quantitative analysis requires subtracting the paired background spectrum from each plasma spectrum. Negative intensities produced after subtraction are clamped to zero to suppress non-physical values arising from detector noise and measurement uncertainty.
    \item \emph{Interactive Visualisation:} Each spectrum contains 2,094 sampled wavelength channels. Efficient web-based visualisation therefore benefits from adaptive downsampling techniques that preserve diagnostically important emission peaks while reducing rendering overhead.
\end{enumerate}

To our knowledge, no publicly documented open-source software currently provides an end-to-end processing workflow specifically for Chandrayaan-3 LIBS Level-1 products. Existing software such as \emph{libpyhat} \cite{anderson2019libpyhat} provides documented capabilities for spectral preprocessing, visualisation, and machine-learning workflows after spectra have been imported into its analysis framework. However, no documented support is provided for native Chandrayaan-3 PDS4 archive ingestion, mission-specific XML metadata extraction, plasma--background pairing using LIBS status flags, automated provenance verification through archive checksums, transformation of wide-format spectra into a normalised relational database, or integrated REST-based data delivery. Consequently, researchers must currently implement substantial mission-specific preprocessing before quantitative analysis can begin.

LunarAtlas addresses this software gap by providing an open-source, deterministic, and reproducible processing infrastructure that automates PDS4 metadata parsing, archive integrity verification, plasma--background pairing, background subtraction, relational database ingestion, REST API generation, and interactive spectral visualisation. By transforming archived Chandrayaan-3 LIBS products into analysis-ready spectral records, LunarAtlas enables reproducible downstream scientific workflows while eliminating the need for custom mission-specific preprocessing software.

\section{Software description}

\subsection{Software Architecture}

LunarAtlas adopts a three-tier architecture. The overall system flow is shown in Figure~\ref{fig:architecture}. The layers consist of:
\begin{itemize}
    \item \textbf{Data Layer:} A PostgreSQL database \cite{postgresql} stores mission metadata, observation provenance, and processed spectra. Processed values are stored in a normalised long-form layout, where each record represents a single channel value.
    \item \textbf{Logic Layer:} Implemented in Python~3.11 and FastAPI \cite{fastapi}, the logic layer manages NIST atomic spectra matching, adaptive downsampling, and API query routing. Result caching is handled via Redis \cite{redis}, which stores recently served downsampled payloads to reduce repeated database round-trips and maintain sub-500~ms API response latency on a single-node workstation.
    \item \textbf{Presentation Layer:} A browser-based React client \cite{react} renders interactive spectral plots. To ensure scientific integrity, all data reduction remains server-side.
\end{itemize}

\subsection{Data Processing and Cleaning Pipeline}

The processing pipeline is implemented in Python across two primary modules: \texttt{step2\_process\_l1\_data.py} (ingestion, pairing, cleaning) and \texttt{step3\_graph\_plotting.py} (QA plot generation). The pipeline is fully deterministic and executes in six sequential stages to transform raw PDS4 Level-1 (L1) products into cleaned relational records:

\begin{enumerate}
    \item \textbf{PDS4 Label Parsing:} The pipeline parses the accompanying PDS4 XML label to extract logical identifiers, observation start/stop times, physical units, and instrument metadata. The parser follows PDS4 field semantics \cite{pds4_standard} to locate mandatory fields including \texttt{pds:logical\_identifier}, \texttt{pds:start\_date\_time}, and \texttt{pds:stop\_date\_time}; labels from which these fields cannot be extracted are skipped with a warning.
    \item \textbf{File Provenance Verification:} A 128-bit MD5 checksum is computed over the raw L1 CSV bytes before any processing begins. MD5 is used for file integrity verification in a trusted archive context, not for cryptographic security. The digest is stored alongside ingestion timestamps and algorithm version strings, establishing a traceable audit trail. The pipeline is fully deterministic: re-running on the same input file produces bit-identical output records.
    \item \textbf{Wide-to-Long Reshaping:} Spectra are distributed as wide CSV tables in which each wavelength channel (164.35--878.26~nm; 2,094 channels total) is represented as a separate column \cite{numpy, pandas}. The pipeline reshapes each row into a normalised long-form layout:
    \begin{equation}
        (\text{Measurement\_ID}, \text{Wavelength\_nm}) \rightarrow \text{Intensity (a.u.)}
    \end{equation}
    This relational representation eliminates position-based column assumptions and supports efficient range-based queries.

\begin{figure*}[!t]
    \centering
    \includegraphics[width=0.9\textwidth]{publication_figures/fig2_reshape_schematic}
    \caption{L1 Wide-Format Column Structure to Normalised Long-Form Schema Ingestion.}
    \label{fig:reshape_schematic}
\end{figure*}

    \item \textbf{Plasma-Background Identification:} The pipeline parses Force Reset Status ($F$) and Laser Fire Status ($L$) metadata flags to identify background ($B$) and plasma ($P$) shots:
    \begin{equation}
        B: (F = 1) \land (L = 0) \quad \text{and} \quad P: (F = 0) \land (L = 1)
    \end{equation}
    Background association relies on acquisition order preserved in the L1 CSV. Each plasma acquisition is associated with the most recently observed valid background acquisition, which is ordered per the PRADAN file specification. Acquisitions satisfying neither condition are considered anomalous and excluded. When consecutive plasma shots appear without an intervening background, each is subtracted from the most recently seen background acquisition.
    \item \textbf{Background Subtraction and Clamping:} Each plasma shot $P_i$ is paired with its most recently preceding background shot $B_i$, under the assumption of measurement stationarity within a single acquisition cycle. The clean spectrum is computed channel-wise:
    \begin{equation}
        I_{clean}(\lambda) = \max(0, I_{P_i}(\lambda) - I_{B_i}(\lambda))
    \end{equation}
    Intensity values represent detector response arbitrary units (a.u.) and must be non-negative; negative values arising from noise fluctuations are set to exactly zero. Acquisitions in which a high fraction of channels are clamped to zero are flagged with a quality warning, as this indicates a likely failed or weak plasma event.
    \item \textbf{Measurement ID Generation:} To resolve timestamp collisions (when multiple acquisitions share a single clock second), the pipeline assigns a sequential, file-unique Measurement ID. IDs are unique within a file; during database ingestion, a globally unique identifier is constructed by combining the observation-specific file identifier with the local Measurement ID, enabling unambiguous retrieval of spectra across the entire archive. This ID groups the 2,094 wavelength-intensity records into a single logical entity.
\end{enumerate}

\subsubsection{Justification of Paired Background Subtraction}
Paired subtraction was selected over model-based baseline methods for three reasons: (i) the LE-LIBS acquisition protocol provides temporally matched background measurements, making paired subtraction directly applicable without additional modelling assumptions \cite{sridhar2024libs}; (ii) the operation is fully deterministic and parameter-free, satisfying the reproducibility requirements of the infrastructure layer; and (iii) downstream users may apply further denoising (e.g., ALS \cite{eilers2003perfect} or Savitzky-Golay \cite{savitzky1964}) without the cleaning tier imposing irreversible smoothing choices. \subsubsection{LTTB+Peaks Adaptive Downsampling}
A full-resolution LE-LIBS spectrum contains 2,094 channels. Rendering thousands of spectra simultaneously in a browser-based visualisation console creates significant client-side overhead. Downsampling is therefore required for interactive visualisation while preserving diagnostically important spectral features.

LunarAtlas downsamples the data to a target count $N(k) = \min(2094, \lceil 2094 \cdot 2^{-k} \rceil)$, where $k$ is the zoom level ($k=0$ returns the full 2,094 points). The Largest Triangle Three Buckets (LTTB) algorithm \cite{steinarsson2013lttb} partitions the spectrum into buckets and selects points that maximise the triangle area formed with adjacent buckets. For a candidate point $P_c$ in bucket $j$, where $i_a, i_c, i_b$ denote the channel indices and $I_a, I_c, I_b$ their respective intensities, the signed triangle area is:
\begin{equation}
    A(P_a, P_c, P_b) = \frac{1}{2} |i_a(I_c - I_b) + i_c(I_b - I_a) + i_b(I_a - I_c)|
\end{equation}
The candidate maximising $|A|$ is selected. However, geometric downsampling algorithms like LTTB do not guarantee peak preservation.

To prevent the loss of narrow diagnostic emission lines, LunarAtlas implements a Targeted Peak Preservation (Peak-Union) step. The Peak-Union procedure operates as follows:
\begin{enumerate}
  \item Retrieve the set of curated NIST reference wavelengths $P_{\text{NIST}}$ corresponding to target elements under investigation (e.g. Ca II at 393.366~nm, Mg II at 279.553~nm, etc.).
  \item Run standard LTTB on the full spectrum to produce the downsampled index set $S_{\text{LTTB}}$.
  \item Compute $S_{\text{final}} = S_{\text{LTTB}} \cup \{i_p : p \in P_{\text{NIST}}\}$, where $i_p$ is the nearest channel index to each reference wavelength.
  \item Sort $S_{\text{final}}$ by wavelength index.
\end{enumerate}
The output size is therefore $|S_{\text{final}}| \leq |S_{\text{LTTB}}| + |P_{\text{NIST}}|$, with equality when no target peaks fall on LTTB-selected indices.

The peak retention rate measures the percentage of originally detected spectral peaks that are preserved in the downsampled subset (either exactly or at adjacent channels).

\section{Illustrative examples}

A typical workflow involves first installing dependencies, starting the storage backends, and executing the multi-stage ingestion pipeline via a shell script. Once processed, the servers can be started and queried programmatically. The following CLI commands initialise the environment, run the pipeline, and boot up both the API server and the frontend visualizer:

\begin{lstlisting}[language=bash, caption=Environment setup, pipeline execution, and server boot commands]
# Install server and client dependencies
pip install -r core/server/requirements.txt
cd core/client && npm install && cd ../..

# Start PostgreSQL and Redis containers
docker-compose up -d db redis

# Execute the 8-stage data processing pipeline
./Pipeline/run_pipeline.sh

# Start the FastAPI backend API server
cd core/server && uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# Start the React client developer server (in a separate shell)
cd core/client && npm run dev
\end{lstlisting}

Once the services are active, the REST API can be queried programmatically. The backend server exposes several endpoints: (i) \texttt{/api/v1/spectrum} which retrieves clean spectral curves with adaptive LTTB+Peaks downsampling and optional denoising filters (ALS, Savitzky-Golay); (ii) public index queries such as \texttt{/api/v1/public/observations} and \texttt{/api/v1/public/measurements} which expose PDS4 metadata and physical parameters (laser energy, pulses) filtered by date or identifier; and (iii) status monitors such as \texttt{/api/v1/health}. 

The following script queries the \texttt{/spectrum} endpoint to retrieve a peak-locked downsampled spectrum for a processed measurement:

\begin{lstlisting}[language=Python, caption=Python client request for downsampled spectra]
import requests

url = "http://localhost:8000/api/v1/spectrum"
params = {
    "measurement_id": "1",  # clean Measurement ID
    "lambda_min": 240.0,
    "lambda_max": 450.0,
    "zoom_level": 2,
    "als": True,
    "savgol": True
}

response = requests.get(url, params=params)
data = response.json()

# Process spectrum points (DownsampledBucket or SpectralDataPoint objects)
for pt in data["data"][:5]:
    if "wavelength_nm" in pt:
        print(f"Wavelength: {pt['wavelength_nm']} nm, Cleaned: {pt['intensity']} a.u.")
    else:
        print(f"Bucket: {pt['lambda_min']} to {pt['lambda_max']} nm")
\end{lstlisting}

\section{Impact}

\subsection{Cleaning Pipeline Validation}

All experimental evaluations and benchmarks in this section were conducted on the Chandrayaan-3 Pragyan Rover LE-LIBS Level-1 (L1) calibrated product \texttt{ch3\_lib\_002\_20230825T104221\_00\_l1} (corresponding to raw file \texttt{ch3\_lib\_002\_20230825T104221\_00\_l1.csv} containing 8 interleaved acquisitions, representing 4 background and 4 plasma measurements).

The LunarAtlas cleaning stage applies two preprocessing operations: (i) subtraction of the most recently acquired background spectrum preceding each plasma shot, and (ii) non-negativity enforcement through zero-clamping. Background subtraction is a standard spectroscopic preprocessing technique used to remove instrumental baseline and continuum contributions prior to spectral analysis \cite{cremers2006libs, chen2023automatic}.

The cleaning pipeline was evaluated on a 20~nm validation window (560--580~nm), a spectral region free of major expected lunar regolith emission lines and representative of the instrumental continuum baseline. In this window, the raw L1 spectrum for Measurement ID~3 had a mean intensity of $\approx1{,}387$~a.u. and a standard deviation of $\approx54$~a.u. In contrast, the cleaned spectrum had a mean intensity of $\approx444$~a.u. and a standard deviation of $\approx71$~a.u. This yields a baseline suppression factor of:
\begin{equation}
    \text{Baseline suppression} = \bar{I}_{raw} / \bar{I}_{clean} \approx 3.1
\end{equation}
The $3.1\times$ suppression demonstrates the removal of the background continuum (Figure~\ref{fig:pre_post_overlay}). The standard deviation increase by a factor of $\approx1.3$ is consistent with noise propagation from paired subtraction \cite{bevington2003data}, closely matching the theoretical $\sqrt{2} \approx 1.4$ noise propagation factor resulting from the subtraction of two independent signals. Prior to clamping, approximately 0.5\% of all spectral channels for Measurement ID~3 contained negative intensity values due to statistical fluctuations. Since detector response intensity values (a.u.) are physically non-negative quantities, all negative values were clamped to zero. \subsection{Downsampling and Peak Preservation}

Evaluation of the downsampling algorithms was performed on valid plasma measurements (IDs 3 and 4) which exhibit clear emission features; Measurements ID~3 and ID~4 were selected because they contain sufficient spectral peaks for quantitative evaluation, whereas low-signal acquisitions (e.g., ID~2) do not permit meaningful peak-preservation analysis. Weak or failed plasma acquisitions (IDs 1 and 2, characterised by $>22$\% clamped channels) were excluded for this reason. Peak detection was performed using \texttt{scipy.signal.find\_peaks} with a prominence threshold of $3\sigma_{\text{noise}}$ (where $\sigma_{\text{noise}}$ is the standard deviation in the 560--580~nm continuum window) and a minimum inter-peak distance of 5 channels. This yielded 76 local maxima in Measurement ID~3 and 91 in Measurement ID~4, with prominence thresholds of $3\sigma = 212.28$~a.u. and $140.65$~a.u. respectively.

The LTTB+Peaks downsampling algorithm was compared against other decimation algorithms at 10\% data density (Table~\ref{table:downsampling}). The Retention (\%) column reports the fraction of 76 detected local maxima in Measurement ID~3 retained by each algorithm; RMSE is computed relative to the full-resolution 2,094-point spectrum.

RMSE quantifies global spectral shape fidelity---how well the downsampled envelope approximates the full-resolution spectrum. Peak retention quantifies feature-level fidelity---whether diagnostically important emission features survive decimation. Both metrics are complementary: an algorithm may achieve low RMSE while discarding narrow peaks, or preserve all peaks while distorting the spectral envelope. While standard LTTB achieves a low RMSE (90.10~a.u.), it only retains 64.5\% of local maxima. The addition of the Peak-Union step in LTTB+Peaks achieves 100\% retention of detected peaks with negligible RMSE penalty (92.25~a.u.).

To validate the peak preservation both from an engineering perspective (all detected peaks) and a spectroscopy-oriented perspective (NIST reference lines), we report a dual retention metric in Table~\ref{table:dual_retention}. A matching tolerance of $\pm0.5$~nm was adopted to accommodate the spectral resolution of the Level-1 dataset and minor wavelength calibration uncertainties. Out of 76 detected local maxima in Measurement ID~3, 5 peaks matched NIST reference wavelengths within this tolerance. \section{Conclusions}

LunarAtlas provides an open-source, reproducible data processing and database infrastructure for Chandrayaan-3 LE-LIBS Level-1 observations. The pipeline implements a deterministic processing pipeline that resolves the challenges of tabular PDS4 parsing, interleaved acquisition pairing, and physically motivated background cleaning---validated through quantitative evaluation of cleaning, peak preservation, and API performance. The peak-locked LTTB downsampling algorithm achieved 100\% peak retention across all evaluated spectra and density levels (5--50\%) during visualisation while maintaining sub-500~ms response times on commodity hardware. By establishing an auditable, versioned reference archive for Chandrayaan-3 LIBS data, LunarAtlas supports quantitative lunar science and offers a template architecture applicable to future planetary LIBS mission archives.

\paragraph{Limitations.} While the LunarAtlas pipeline successfully processes all available Level-1 observation products, the benchmarks and evaluations reported in Section 4 were performed on a single representative reference product (\texttt{ch3\_lib\_002\_20230825T104221\_00\_l1}) containing 4 cleaned spectral measurements. Statistical claims about inter-acquisition variability are therefore preliminary and should be extended as additional mission data become available. The statistical evaluations reported in Section 4 should therefore be interpreted as software validation rather than a comprehensive analysis of lunar LIBS measurements. The system has been validated on a single-node configuration and has not been benchmarked under concurrent multi-user load. No Level-2 calibration (e.g., elemental abundance quantification) is implemented; LunarAtlas provides analysis-ready L1 records as input to such workflows.

\paragraph{Future work.} Planned extensions include: (i) ingestion of additional Chandrayaan-3 LE-LIBS observation files as they are released through PRADAN; (ii) integration with Level-2 calibration workflows including automatic elemental abundance estimation; (iii) multi-user concurrent query support via horizontal scaling; and (iv) a containerised one-command deployment via \texttt{docker-compose}.
